MÉTHODEComment ces tests ont été menés

Approche end-to-end, à travers les vraies interfaces de l'application : routes HTTP REST, endpoint TestControlServer, et rendu HTML/JS via Chrome headless.

Outils utilisés

  • curl — pour toutes les requêtes HTTP (GET / POST / PUT / DELETE) contre l'API REST http://127.0.0.1:8444/api/*
  • jq — pour extraire et formater les payloads JSON capturés
  • Google Chrome headless — pour capturer les screenshots des pages HTML servies (1400 × 900 à 1400 × 5500 selon le contenu)
  • Bash — pour orchestrer les tests et capturer les codes HTTP, tailles de réponse, temps d'exécution

Périmètre testé

  • Toutes les routes REST documentées dans BrApiRouter exposées en production
  • Endpoint TestControlServer POST /api/command pour les commandes métier (tickets, recipes, cooking)
  • Endpoints publics (sans authentification) : QR consultation carte, documents HTML
  • Cas d'erreur attendus : PIN incorrect, camera sans secret, dépendance de feature flag non remplie
  • Mutations d'état : création ingrédient / perte / carte / catégorie / item / table / feature flag
  • Rendu HTML des 6 documents servis

Le binaire br_server en cours d'exécution a été compilé depuis la branche main au commit cdd2e7b. La base SQLite utilisée est ~/.broccers/broc.db, en mode WAL, partagée avec la prod.

Distribution : Le bundle .app peut être copié sur n'importe quel Mac via AirDrop, USB ou DMG. Pour distribution App Store ou signature notarisée, prévoir l'enrôlement Apple Developer (~99 €/an) — pas nécessaire pour usage interne.

Pour la connexion au serveur distant (Mac Studio via Tailscale), l'utilisateur doit passer l'URL via --dart-define=BR_API_URL=http://100.95.200.28:8444 au build, ou ajouter un écran de configuration. À traiter en v0.9.

FINDINGSObservations

Le mode est correctement auto-détecté. GET /api/system/config renvoie mode: "tailscale_native" dans cet environnement de test (Mac Studio sur Tailscale). Le détecteur passerait à docker ou usb_portable selon les variables d'environnement / présence de /.dockerenv.
22 / 30 feature flags ON par défaut. Les 8 OFF correspondent à : 4 caméras ESP32 (Phase H), 1 commande client au QR (Phase J), 1 réservations (Phase K), 1 multi-instance, 1 branding personnalisé, 1 rate-limit strict. Note : le flag camera.face_recognition est temporairement ON dans cet environnement car activé manuellement pour les tests camera-events.
Toutes les mutations sont audit-loggées. Les 20 derniers événements du journal incluent les actions de ce rapport (création ingrédient, déclaration waste, création carte, toggle feature flag, événements caméra). Le journal est cohérent et complet.
Le briefing matinal renvoie 500 en mode automatisé. Probablement parce que le subprocess claude n'a pas de TTY interactif disponible. À mitiger pour les tests CI : soit mocker la route, soit utiliser un flag --no-tty côté claude, soit prévoir une erreur gracieuse côté serveur. Hors test automatisé, la feature fonctionne (testée précédemment depuis l'app).
Le screenshot Flutter authentifié n'a pas pu être capturé en headless pur. Chrome headless seul ne préserve pas le localStorage entre deux lancements via --user-data-dir de manière fiable pour les SPA. Pour les futurs rapports, prévoir Playwright qui gère bien l'injection de localStorage + l'attente de boot Flutter (~5-8 secondes). Les écrans Flutter restent visualisables via l'app à http://100.95.200.28:8766/ (PIN 1234).
Le rendu QR offline est confirmé. La consultation publique /menu/<id>/<secret> rend correctement les couleurs Broc, le brand CAFÉ, la version de carte, et tous les items disponibles. Aucun appel à un service externe détecté (Chrome charge la page sans erreur de CSP).
Mode 86 limité aux cartes publiées. Comportement par design (ToggleMenuItemAvailabilityUseCase ne touche que la carte courante publiée). Évite les ruptures « fantômes » sur une carte invisible côté client. À documenter dans la spec si le sujet revient.

VERDICTConclusion

Les 11 modules métier de Broccers v0.8.0 sont opérationnels sur le périmètre testable côté serveur (REST + TestControlServer + endpoints publics). Sur 32 tests exécutés (21 happy-path + 11 probes adversariaux), 31 passent comme attendu. Le seul résultat divergent (Mode 86 sur carte draft) est un comportement métier voulu, à documenter et non un bug.

Sur les 6 documents HTML servis, tous renvoient HTTP 200 avec une taille cohérente (16 Ko à 97 Ko) et se rendent visuellement sans erreur dans Chrome headless. Les 6 captures écran ci-dessus le prouvent.

L'endpoint ESP32-S3 Sense (Phase H caméras IA) est opérationnel côté backend : refuse 401 sans secret, refuse 403 quand feature flag OFF, accepte 201 quand secret + flag ON. Le firmware côté ESP32 reste à coder mais l'ingestion est prête à recevoir.

L'application Broccers v0.8.0 telle que livrée sur GitHub (commit cdd2e7b) est fonctionnelle pour ouverture du Broc Café. Les findings ci-dessus sont à traiter en améliorations continues, pas en blocants.

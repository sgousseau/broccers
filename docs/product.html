02 Features livrées

10 onglets UI couvrant tout le cycle d'exploitation d'une brasserie : personnel & paie · cuisine vocale · carte imprimable · courses · IA · audit · coûts · pertes · tables QR · paramètres.

v0.1.0 · 4 features fondatrices

🧱 Bootstrap

Personnel (CRUD + planning + clock-in/out + pauses droit du travail FR), carte dynamique avec PDF A4 imprimable allergènes INCO, listes de courses checklist par fournisseur, chat IA Claude avec contexte Broc.

PHASE A · multi-rôles dynamique

👥 Multi-rôles + Journal

SgEmployee.roles: Set (capabilities) + weeklyDefault (planning hebdo) + SgShiftSegment (N segments par shift) + SgEventJournalEntry (audit log universel). Eros peut changer entre runner/barman/plonge en cours de shift, tout est tracé avec actor + reason.

PHASE B · coûts

💶 Taux horaires + Conso staff

SgHourlyRate versionné par employé/rôle (history), SgStaffConsumption (repas/boissons consommés par staff, paid ou à débiter). ComputeShiftCostUseCase agrège heures × tarifs par segment pour total main d'œuvre.

PHASE D · IA + analyse

☀️ Briefing matinal + Pourboires + Heatmap + Onboarding

Bouton AppBar génère briefing Claude (planning + plats + courses urgentes). tipCents sur chaque shift. Heatmap 7j employé × jour. Onboarding par rôle : Claude génère 10-15 items concrets (HACCP-aware) cochables.

PHASE E · cuisine vocale

🎙️ Commande vocale → Tickets cuisine

Le serveur dicte « table 7 deux ricards une entrecôte saignante avec frites un magret rosé sans gratin ». Whisper STT (optionnel) → Claude parse JSON → SgKitchenTicket draft avec items + modifiers. Validation visuelle → envoi cuisine.

PHASE E · cuisine

🍳 Recettes / Étapes / Cuissons

SgRecipe par menu item avec SgRecipeStep (type : prep/cooking/plating/rest, durée attendue). Au send-to-kitchen, SgCookingTask auto-créées. Cuisine clique Démarrer → timer. ⚠ DÉPASSÉ en rouge si overdue.

PHASE F · pilotage économique

⚙️ Paramètres évolutifs (manager-only)

Seuils marge brute ROUGE/JAUNE évolutifs (par défaut 60/70 — ajustables selon conjoncture), profil low_cost/standard/premium, taux de charges FR (66 % par défaut — taxes/charges sociales), seuil d'alerte gaspillage hebdo, configs commande vocale. Toute modification est audit-loggée (setting.changed dans event journal).

PHASE F · composition

📦 Ingrédients + Recettes + Marges

SgIngredient (nom, unité, prix moyen courant, fournisseur). SgRecipeIngredient lie une recette à ses ingrédients avec quantités. ComputeMenuItemCostUseCase calcule coût matière + marge en temps réel. Code couleur ROUGE/JAUNE/VERT selon les seuils paramétrables. Substitution d'ingrédient (livraison non conforme) avec recalcul automatique de marge et tag [SUB].

PHASE F · pertes

🗑️ Tracker gaspillage (observability bienveillante)

SgFoodWaste : kind (ingredient/menuItem), quantité, raison (prepLoss · servingLoss · expiry · unusedProduction · staffMeal · customerReturn · burnt · contamination · other), valeur estimée. Auto-évaluation du coût si l'ingrédient est référencé. Vue récap 7j : total €, breakdown par raison + par jour. Le journal sert à comprendre et améliorer, jamais réprimander.

PHASE F · disponibilité

🚫 Mode 86 / Rupture item

Toggle inline depuis l'onglet Carte : « plus de saumon, livraison demain ». La raison est enregistrée. La carte publique consultable au QR se met à jour automatiquement (item barré et tagué EN RUPTURE). L'audit log conserve menu_item.unavailable.

PHASE F · client-facing

📱 QR codes tables (consultation carte)

SgTable par table avec qrSecret unique. Endpoint GET /menu/<tableId>/<secret> sert la carte publiée en HTML responsive aux couleurs Broc — sans auth, lecture seule. Régénération du secret possible pour invalider l'ancien QR. L'écran Tables permet de créer/désactiver/imprimer.

03 Architecture SG

Clean Architecture 4 couches · Result<T,E> · Port/Adapter universel · Sg prefix obligatoire.

Stack technique

  • Frontend : Flutter Web PWA (br_web), Material + wrappers br_sg.dart (skin Broc, API-compatible sg_ui)
  • Backend : Dart shelf (br_server) sur Mac Studio, port 8444
  • Storage : SQLite WAL local (~/.broccers/broc.db)
  • PDF : package pdf Dart natif (server-side, A4 imprimable)
  • IA : Claude Max via claude -p subprocess sur cluster Tailscale (jamais API Anthropic)
  • STT : Whisper sur Titan via Tailscale (optionnel, fallback texte)
  • Réseau : Tailscale mesh only — aucun cloud public
  • Auth : PIN bcrypt + rate-limit 5/15min + JWT HS256 24h

05 Philosophie SG inviolable

Observability bienveillante. Le journal sert à COMPRENDRE et AMÉLIORER, jamais à réprimander. Les alertes vont au manager, jamais à l'employé directement. Formulation hypothétique (« semble », « peut-être »), pas accusatoire. Objectif : analyse comportementale pour optimiser, pas surveillance.

Règles SG inviolables (CLAUDE.md projet)

  • Clean Architecture 4 couches (Domain / Adapter / Provider / UI)
  • Result<T,E> partout — jamais de throw cross-layer
  • Port / Adapter pour toute dépendance externe (Claude, SQLite, Whisper, PDF, auth)
  • Sg prefix sur toute classe/enum publique
  • sg_ui obligatoire dans br_web (via wrappers br_sg.dart en attendant migration complète sg_ui v0.4)
  • Command-First : chaque feature DOIT exister en CLI/TestControlServer avant l'UI
  • CHANGELOG.md de chaque package à chaque changement
  • SDK uniforme >=3.8.0 <4.0.0
  • Pas d'API Anthropic directe — toujours claude -p cluster Tailscale
  • Pas de quick fix — toute problème récurrent = abstraction Port/Adapter
